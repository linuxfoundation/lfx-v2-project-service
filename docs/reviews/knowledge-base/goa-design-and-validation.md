# Goa design and validation

Patterns specific to this repo's Goa design files under
`api/project/v1/design/` and the generated boundary under `api/project/v1/gen/`.
The recurring theme is field validation: optional string fields that persist as
`""` must accept the empty string on the way back in, or GET→PUT round-trips and
client response validation fail. There is also a hard generated-code boundary
the team enforces, and an authorization coupling between design changes and the
Helm ruleset.

**Read when:** any file under `api/project/v1/design/**` or
`api/project/v1/gen/**` changed, or `cmd/project-api/service_endpoint_*.go`
changed, or `charts/lfx-v2-project-service/templates/ruleset.yaml` changed.

---

## `goa-design-and-validation/format-rejects-empty-string` — Important

**Pattern:** an optional `UserInfo` field (avatar, email) — or any optional string field that the service may persist as `""` — is constrained with `Format(FormatURI)` / `Format(FormatEmail)`. The service stores an omitted value as an empty string, then a GET→PUT round-trip (or a client response decode) fails validation because `""` is not a valid URI/email.

**Detect:** in `api/project/v1/design/types.go` (and other design files), find optional fields using `Format(FormatURI)` / `Format(FormatEmail)` / `Format(FormatUUID)` on a field that can legitimately be empty. The repo's fix is a regex `Pattern` that allows `""` OR the real format, e.g. `Pattern(`^$|^[a-zA-Z0-9._%+\-]+@...$`)`.

**Empirical citation:** PR #53 (avatar) — "Replaced `Format(FormatURI)` with a regex `Pattern` for `UserInfo.avatar` to allow `""` or a URI-like string." PR #63 (email) — "updates the `UserInfo.email` validation rules so that an empty string (`""`) is accepted ... preventing GET→PUT round-trip failures when the service persists an omitted email as an empty string." Recurred at PR #71 and is the live convention: `api/project/v1/design/types.go` uses `Pattern(`^$|...`)` for email and avatar.

**Failure message:** Optional field uses `Format(...)` that rejects the empty string — GET→PUT round-trip or client response validation fails when the value is persisted as `""`.

**Fix:** replace the `Format(...)` constraint with a regex `Pattern` that allows `""` or the real format (`^$|<real-pattern>`), then run `make apigen` and commit the regenerated `gen/` files (including OpenAPI specs) in the same change.

## `goa-design-and-validation/missing-format-enforcement` — Important

**Pattern:** a field that has a known format (a UUID like `parent_uid`, the project slug) is declared as a plain string in the design with no `Format`/`Pattern`, pushing validation into hand-written service code or leaving it unenforced.

**Detect:** in design types, flag UID/slug/identifier string fields that lack `Format(FormatUUID)` or a `Pattern`. The repo enforces `Format(FormatUUID)` for UUIDs and `Pattern(`^[a-z][a-z0-9_\-]*[a-z0-9]$`)` for the project slug.

**Empirical citation:** PR #1 `cmd/project-api/design/types.go:16` (jordane) — "Any reason not to [enforce this] as a UUID right now?" and `:41` — "We can enforce a format on this [via regexp]." Acted on; `api/project/v1/design/types.go` now carries `Format(FormatUUID)` and `Pattern(`^[a-z][a-z0-9_\-]*[a-z0-9]$`)`.

**Failure message:** Identifier/slug field declared without `Format`/`Pattern` — validation is left to hand-written code or skipped.

**Fix:** add `Format(FormatUUID)` for UUID fields and the established slug `Pattern` in the design, regenerate, and commit the `gen/` diff. (Note: `parent_uid` deliberately omits `Format(FormatUUID)` so the server can also accept the empty string — see the design comment — so allow the empty-string-or-UUID variant there.)

## `goa-design-and-validation/hand-edited-generated-code` — Critical

**Pattern:** a finding or change touches a file under `api/project/v1/gen/**` directly, or proposes editing generated code. Generated files are tracked but produced by `make apigen` and must never be hand-edited; the next `goa gen` overwrites them.

**Detect:** any hand edit to a file under `api/project/v1/gen/**` that is not the byproduct of a design change + `make apigen`. Also flag review suggestions that ask to edit generated server/client/CLI code.

**Empirical citation:** PR #16 `api/project/v1/gen/http/project_service/server/server.go:618` (jordane) — "This is code generated by goa and should not be modified directly." Also PR #1 `cmd/project-api/gen/http/cli/project_service/cli.go:187` — a suggested edit was withdrawn once the reviewer noticed "it is from the gen folder."

**Failure message:** Generated code under `api/project/v1/gen/` edited by hand — it will be overwritten by the next `make apigen`.

**Fix:** change the source design under `api/project/v1/design/`, run `make apigen`, and commit the regenerated output. Never edit `gen/` directly.

## `goa-design-and-validation/auth-change-without-ruleset` — Critical

**Pattern:** a design change adds or alters an endpoint's authorization (new route, changed method, changed required relation) without updating `charts/lfx-v2-project-service/templates/ruleset.yaml` in the same PR. The Heimdall ruleset and the API then disagree on who may call the endpoint.

**Detect:** when `api/project/v1/design/**` changes add/modify a route or its auth, confirm `charts/lfx-v2-project-service/templates/ruleset.yaml` is changed in the same diff with a matching rule. Also confirm the new rule's OpenFGA `object`/`relation` is correct (and that `parent_uid` is validated where the rule reads `.Request.Body.parent_uid`).

**Empirical citation:** repo convention recorded as a CodeRabbit learning on this repo — "Applies to charts/*/templates/ruleset.yaml : Update Heimdall ruleset in `charts/*/templates/ruleset.yaml` when adding new endpoints" (surfaced in PR #10). Reinforced by PR #21 `charts/.../templates/ruleset.yaml:49` (Copilot) flagging the `parent_uid` authorization check, and stated in `CLAUDE.md`: "any design change that affects authorization must also update `charts/lfx-v2-project-service/templates/ruleset.yaml` in the same PR."

**Failure message:** Authorization-affecting design change without a matching `ruleset.yaml` update — Heimdall and the API disagree on access.

**Fix:** add/update the corresponding rule in `charts/lfx-v2-project-service/templates/ruleset.yaml` in the same PR, with the correct OpenFGA object/relation. Validate parent-based rules handle a missing/empty `parent_uid` (the project service maps that to 403, not a bypass).
